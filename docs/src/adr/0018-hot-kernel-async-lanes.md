# ADR-0018: Hot kernel + async lanes

- **Status:** accepted
- **Date:** 2026-05-17
- **Deciders:** ActantDB Authors

## Context

ActantDB now spans 50+ work packages and 40+ crates covering Chronicle, Commands, Effects, Guard, Memory, Context, Workflows, Replay, Index, Embedders, Prompts, Models, Cache, Trace, Protocol, Throttle, Circuit, Lock, Ingress, Sync, Audit Export, and the CLI surface. If every command synchronously touched every relevant subsystem, even basic interactions would be slow:

- `append_user_message` → 2 ms or 200 ms?
- `request_tool_call` → 5 ms or 1.5 s?
- `complete_effect` → 3 ms or several seconds?

The agent experience depends on the answer. A 2-second pause before a tool call appears is unacceptable; a 50 ms one is fine.

## Decision

ActantDB is structured as a **hot kernel + asynchronous lanes**:

- The kernel (`actant-kernel`) is the only crate that runs in the synchronous command path. Its job per command is exhaustively defined in `/specs/19-performance-architecture.md` §4:
  1. authenticate actor,
  2. validate command schema,
  3. compiled policy check,
  4. lightweight budget/rate check,
  5. append event,
  6. update hot projection,
  7. enqueue effect (if needed),
  8. notify subscribers.
- Everything else — embeddings, reranking, model calls, workflow advancement, memory candidate generation, eval shadow, OTel export, compliance evidence, graph extraction, artifact processing, audit rollup — runs as **lane consumers** of the chronicle. Lanes subscribe to events, do their work, and emit follow-on events.

Latency budgets are explicit (`/specs/19-performance-architecture.md` §2). A `bench/` harness enforces them in CI; regressions > 10 % fail the PR.

## Consequences

### Positive

- Commands stay short and predictable. The agent feels responsive.
- Adding new capability layers (a new reranker, a new compliance pass, a new analytics projection) doesn't slow existing commands — they ship as lanes.
- The kernel is the **only** place to look for hot-path performance regressions. Search radius for "why is this slow?" stays bounded.
- The lane pattern is the natural extension of the Subscription Engine; we already build on it.
- Replay is sharper: the kernel's contribution is deterministic; lane outputs are recorded effect results that replay re-uses.

### Negative

- Two-phase mental model. New contributors must internalize "this work goes in the kernel; that work goes in a lane." Mitigated by the per-operation sync/async breakdown in `/specs/19-performance-architecture.md` §5 and by the grep architectural tests.
- Eventual consistency for non-critical reads. A memory candidate generated by a lane is not visible the instant the triggering event commits. Mitigated by minimal-facts-now-richness-later (ADR-0019) and by Studio surfaces showing both states.
- Lane backpressure is real and must be managed. Mitigated by `/specs/19-performance-architecture.md` §9 backpressure signals and explicit admission control.

### Neutral / open

- Whether a "synchronous read-your-writes" mode for lanes is needed in dev mode is unresolved. Phase 2 may add a `wait_for_lane(lane, predicate)` test helper.

## Alternatives considered

- **Synchronous everything.** Rejected — caps the system's growth at the slowest subsystem.
- **Two services (fast core + slow workers) connected by message bus.** Functionally similar to the kernel + lanes pattern; we chose to keep it in-process per deployment with the Subscription Engine as the bus, so local mode stays simple.
- **Optional async opt-in per feature.** Rejected — defaults shape behavior; the default must be the right answer.
