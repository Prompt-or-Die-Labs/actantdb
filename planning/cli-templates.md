# Template catalog

Each template scaffolds a working ActantDB project that runs end-to-end with `actant dev`. Templates are stored at the repo root under `/templates/<name>/` and embedded in the CLI binary by `actant-templates`. Each has its own README inside.

A template's job is to make the first ten minutes of a developer's interaction with ActantDB feel inevitable.

## Templates

### `minimal`

The smallest functional ActantDB project. Two commands, one schema file, no workers beyond the embedded model worker stub.

Files:

```
my-agent/
├── actant.yaml                       (single workspace, sqlite, port 4739)
├── schema/main.actant
├── commands/append_message.py        (or .ts)
└── examples/hello.py
```

Use for: documentation, smoke tests, "what does ActantDB look like at minimum?"

Ships in: **Phase 1**.

### `coding-agent`

The flagship template. The alpha demo materialized as a template.

Files:

```
my-coding-agent/
├── actant.yaml
├── schema/
│   ├── tools.actant                  (shell, file)
│   └── memory.actant
├── commands/
│   ├── approve_tool_call.py
│   └── record_tool_result.py
├── workers/
│   ├── model_worker.py
│   ├── shell_worker.py
│   └── file_worker.py
├── agents/coding_agent.py
├── workflows/fix_tests.actant
├── evals/fix_tests_eval.yaml
├── examples/first_session.py
└── tests/test_memory.py
```

Features: shell + file tools, approval flow, memory review, replay demo, one eval.

Ships in: **Phase 1**.

## Template versioning

Each template has a `template.yaml`:

```yaml
name: coding-agent
version: 1
min_cli_version: 0.1.0
languages: [python, typescript]
features: [memory, approvals, shell, file, replay, evals]
phase_ships_in: 1
```

The scaffolded project records the template version in `actant.yaml` so future `actant upgrade` can apply template-aware migrations.

## Template content rules

- A template MUST scaffold a project that `actant dev` boots successfully on the developer's first run.
- A template MUST include an `examples/` directory with at least one runnable script.
- A template MUST work with `actant doctor` clean on a fresh install.
- A template MUST NOT contain secrets — only placeholders + `.env.example`.
- A template's `README.md` MUST end with a "Try" section listing the next three commands.

## Adding a template

1. Create `/templates/<name>/` with the file tree.
2. Add a `template.yaml`.
3. Add an entry to this catalog.
4. Add an integration test under `crates/actant-cli/tests/templates_<name>.rs` that scaffolds + `actant dev`s the template and asserts success.
5. Update `cli-design.md` § "CLI staging across phases" if the template ships in a specific phase.
