# ADR-0009: The `.actant` schema DSL

- **Status:** accepted
- **Date:** 2026-05-17
- **Deciders:** ActantDB Authors

## Context

ActantDB has a fixed core schema (`/migrations/0001_initial.sql` + `/migrations/0002_extended_primitives.sql`). Developers building agents on top of ActantDB need to declare their own projection tables, project-level commands, and workflows.

The candidate formats:

1. **Raw SQL + hand-written commands.** Familiar but verbose. Developers maintain three things in lockstep (migration SQL, Rust/Python types, command logic) and the connection between them is fragile.
2. **YAML config.** Easier to write than SQL but loses expressive structure (no first-class `command` block; commands are described as YAML lists with conventions instead of grammar).
3. **A small DSL** that expresses both tables and commands, and compiles to SQL + typed code + command stubs.

The Actant Contract makes (3) compelling: a `command` block is a *semantic* object with input schema, required authority, and emitted events. Forcing developers to glue these together by convention introduces drift; declaring them in one grammar prevents drift by construction.

Prior art: Prisma's `schema.prisma`, Convex schema modules, Hasura metadata.

## Decision

ActantDB ships a `.actant` schema DSL, implemented in `actant-schema-dsl`. The DSL covers:

- `table <Name> { field: Type modifier* }`
- `command <Name> { input { ... } requires <permission> emits <event> }`
- `workflow <Name> { node+ edge+ }`
- `enum <Name> { value, value, ... }`

The DSL compiles to:

- **SQL** — `migrations/NNNN_<diff>.sql` based on a diff against the prior schema state. Used by `actant schema apply`.
- **Rust types** — emitted into the project's chosen crate (or skipped if the project is non-Rust).
- **Per-language stubs** — Python (Pydantic), TypeScript (Zod/TS types), Swift (Codable).
- **Command skeletons** — generated by `actant generate command <name>`.

The type vocabulary is closed for v1: `ID`, `String`, `Int`, `Float`, `Bool`, `Timestamp`, `JSON`, `[T]`, `T?`, plus the closed enums `Sensitivity`, `Visibility`, `RiskLevel`, `ReviewStatus`. Developers may declare custom `enum` decls but cannot shadow built-ins.

## Consequences

### Positive

- One declaration → SQL + types + stubs. Drift between them is structurally prevented.
- Commands declared in the DSL get a typed input schema, automatic JSON validation, and uniform error mapping.
- Diff semantics enable `actant schema diff` and `actant schema migrate` — developer-friendly migrations without writing SQL by hand.
- The DSL is small enough to learn in an hour and read at a glance.
- IDE support is straightforward (LSP plugin, Phase 3+).

### Negative

- A new language to learn. Mitigated: the DSL is small and the cheat sheet fits on one page.
- Migrations across DSL versions need rules. Mitigated: explicit `@retired_in_version` and `@renamed_from` annotations; ambiguous renames are rejected.
- The compiler can produce SQL that the developer didn't expect. Mitigated: `actant schema diff` shows the generated SQL before applying, and `--dry-run` is supported.
- New language fragmentation in the agent-tooling world. Mitigated: we offer a `--target sql` exporter so developers can leave the DSL behind without losing schema continuity.

### Neutral / open

- Whether to support `view`, `index`, or `trigger` declarations is deferred to v2.
- Whether to allow inline policy declarations attached to commands is deferred.
- LSP server is post-Phase 4.

## Alternatives considered

- **YAML-only.** Cleaner to bootstrap but loses the `command` block's structural meaning; developers reinvent it per project. Rejected.
- **Raw SQL + decorators.** Pythonic, but ties developers to one language and re-introduces the drift problem. Rejected.
- **TypeScript-as-config (Convex pattern).** Compelling for TS-heavy shops; alienates Python and Swift. Rejected for v1; reconsidered if a TypeScript-first variant is needed in Phase 6.
- **Generate everything from existing TS/Python types via runtime introspection.** Works for single-language ecosystems; ActantDB is multi-language by design. Rejected.
