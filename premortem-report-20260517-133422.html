ActantDB premortem

Frame: November 17, 2026 — the project has died, wide developer adoption did not happen. 2026-05-17 / premortem run.

Executive summary

  • Most likely failure: F1 + F4 — the market crystallized around Mastra (TS), LangGraph (enterprise), Convex (reactive backend + agent memory) before ActantDB shipped one line of code, and the 5-minute demo that could've cut through won't be ready by month 6.
  • Most dangerous failure: F8 — bandwidth split with Swoosh; determines which product survives. If Wes picks wrong, both die.
  • Most fragile assumption: One person can ship the full plan (40 crates, 4 SDKs, Studio, six modes, full reliability + AI-native + protocols + observability + hot-kernel) in 6 months against funded incumbents with production logos.
  • Most important change: Pick one wedge that is winnable; ship it as a complement to incumbents, not a replacement. Strong candidate: chronicle + replay as a Mastra/Convex plugin distributed via npm.
  • Biggest unresolved unknown: Have you actually talked to 15 working agent developers about whether they'd pay attention to this? No — and the answer almost certainly reshapes the plan.

Hidden assumption. You can ship ActantDB-as-designed, with the team and time you have, against the competitors who already shipped, and earn wide developer adoption.

The architecture isn't wrong. The bet that this version of it — one person, 40 Rust crates, 4 SDKs, splitting attention with Swoosh, six-month horizon, into a TS-volume market that already named its winners — can win wide developer adoption is what's failing. Test it explicitly: take the plan to 15 working agent developers. Ask "what would have to be true for you to move from your current tool to ActantDB?" If fewer than 3 give a concrete answer, the wedge is wrong.

F2 — TypeScript-first market vs Rust-first project

L HighS HighD EasyC High
Story

Launch produces 1.8k r/rust stars; curve flattens. The TS audience asks the same two questions: "why a separate server?" and "can I just npm install this?" Mastra closes the durability gap with an in-process SQLite-backed memory adapter. ActantDB demands a Rust toolchain on CI, a worker process diagram, and a port to expose. 94 % of issues are SDK bugs; 89 % of PRs are Rust core changes from three maintainers. Two of three burn out. November: 400 weekly npm downloads vs Mastra's 600k. The project wasn't killed — it was simply skipped.

Root assumptionTS/Python devs accept a separate Rust server process if the SDK ergonomics are good enough.
Warning signsSDK-reported-issues : SDK-authored-fixes > 5:1 in 60 days · first-time TS contributors open zero PRs against the Rust core after 90 days · "why not just Mastra?" appears unanswered > 3×/week.
Validation test50-person unmoderated onboarding study with TS devs new to Rust. If median time-to-first-working-agent > 2× Mastra's, or > 40 % abandon at "install the server," positioning is broken.
MitigationEmbedded mode: compile core to napi-rs Node addon + WASM. Distribute as @actantdb/core. Server mode becomes scale-out, not default.
Kill criterionIf 6-month weekly npm downloads < 10 % of Mastra's and outside-contributor Rust-core PRs < 5/month, pivot to a TS-native control plane or sunset.

F3 — Spec-to-code gap via coding agents

L HighS HighD ModerateC High
Story

First 8 work packages land in two weeks. Pattern feels validated. By August, 30+ crates have green tests locally. Each agent session has re-derived its own interpretation of "AccountableAction," the audit-log schema, and the error type hierarchy. First workspace-level cargo build in September fails in 312 places. Reconciling actant-core-types ↔ actant-policy invalidates actant-runtime (4,000 lines of agent-generated glue assuming the old shape). Each reconciliation takes 2–3 weeks because Wes didn't write any of it. October: actant new my-agent compiles but panics on first action. The demo never ran end-to-end.

Root assumptionA single human can hold workspace-wide semantic coherence across 40 crates built in parallel by stateless agents from prose specs.
Warning signsFirst cross-crate integration needs type adapters · two agent sessions on the same spec produce materially different trait signatures · open PR queue > 5 for more than a week.
Validation testImplement crates 1–5 with the agent. Have a fresh agent session implement crate 6 using only the published interfaces of 1–5. If > 1 round of interface revision needed, the model is broken.
MitigationReplace prose specs with one machine-checked actant-contracts crate. Every trait + error + schema type lives there once; cargo check is the source of truth.
Kill criterionIf by Aug 1, 2026 cargo build --workspace doesn't succeed with 15+ crates integrated and one end-to-end smoke test passing, abandon 40-crate architecture and rebuild as a 3-crate monolith.

F4 — No working 5-minute demo by month 6

L HighS CriticalD ModerateC High
Story

Months 1–3 feel productive — twelve crates compile, all green. Then the integration tax arrives. Storage was query-pull; subscribe assumed push-from-storage. CLI's actant dev wants templates + schema-dsl + codegen-project + server to interop cleanly; codegen emits modules templates don't reference. Index↔embedders↔cache triangle has three plausible ownership models, no decision. Coding agents are excellent at spec'd seams; bad at deciding which seam is correct when crates disagree. Month 6: actant dev crashes half the time; "live memory candidates" never rendered real data; every blog post ends in future tense.

Root assumptionA working demo is the sum of working crates, rather than the product of integration decisions a human must make in advance.
Warning signsTwo "finished" crates need re-opening to make a third compile · the happy-path script doesn't exist as an executable test by month 2 · any README ends with "this is what it will be."
Validation testWrite a failing end-to-end test in week 1: actant new demo && actant dev boots, accepts one command, fires one approval, surfaces one memory candidate. Crates can be stubs; wiring must be real.
MitigationInvert build order. Week 1: demo as a hardcoded shell with all crates as todo!() stubs satisfying a frozen integration contract. Then fill stubs.
Kill criterionIf at month 4 actant dev can't run the demo end-to-end with stub data, cut scope to a single-binary monolith and ship that as v0.1, or stop.

Revised plan

Change now

  • Reframe the project. Pick one wedge: chronicle + replay, governance plug-in for Mastra/Convex, or local-first private memory for Swoosh. Make the rest later.
  • 40-crate scope freeze. Move 36 crates to future/. Phase 1 = 4 crates: actant-core, actant-eventlog, actant-sdk-ts, actant-cli-minimal. Six weeks.
  • Stop writing specs. Move existing spec set to archive/. The single living spec becomes the actant-contracts Rust crate.
  • Ship demo-first scaffold: a failing end-to-end actant new demo && actant dev test with todo!() stubs, before any feature implementation.

Test before committing

  • 15 customer-discovery interviews this week with working agent developers. One question: "What would have to be true for you to use ActantDB?"
  • 2-week time audit: log every hour against {ActantDB-build, ActantDB-review, Swoosh-build, Swoosh-support}. If ActantDB-direct < 25h/week, fix the structure before writing crate #2.
  • Coding-agent coherence test: implement crates 1–5 with Claude Code; have a fresh session implement crate 6 using only published interfaces. > 1 round of interface revision → replace prose specs with the contracts crate.

Monitor during execution

  • PR review latency on ActantDB. Hard threshold: 72 hours.
  • Commit ratio between ActantDB and Swoosh. Hard threshold: 70/30 in either direction for 3+ weeks triggers a decision.
  • actant new && actant dev cold-start success rate. Goal: 100 % by end of month 2.
  • "Why not just Mastra + Convex?" appearances. Unanswered 3+ times → positioning broken.

Accept as risk

  • The architectural depth may never be fully delivered in v0.1. v0.1 is much smaller than the design; the design becomes the v2 roadmap.
  • Some planning work is "wasted" in the narrow sense. It's also the strongest competitive moat — but only if a v0.1 ships to put it in front of developers.
